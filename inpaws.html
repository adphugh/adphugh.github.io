Indiana Native Plant and Wildflower Society (INPAWS)

Created in Spring 2018 with the ServeIT Clinic
Team members: Sarah Burns, Maritza Castaneda, Karly Kandel, JY Kim, Lucas Rutherford, Caleb Schaftlein, and Cassidy Wichowsky

Scope: Create a web application where users can search for native Indiana plants via garden criteria or by plant name, thus returning native plant pollinators to Indiana.

Technical assets used:
  • PHP
  • MySQL
  • HTML5
  • CSS3
  • Adobe XD
  • WordPress
  • Git
  • JIRA (with agile methodologies)

Background: INPAWS is a nonprofit located in Bloomington, Indiana whose goal is to remove invasive plants from Indiana, promote the planting of native plants, and bring more native plant pollinators back to Indiana. INPAWS found that a lot of amateur gardeners do not know what plants are native to Indiana or which plants bring in certain pollinators. In order to combat this learning curve, INPAWS asked my team to develop a web application where users can list their garden's criteria and receive a list of native plants that fit their garden's characteristics. More advanced gardeners can also search for plants by latin or common name.

My Role: As the project manager on this team, my roles were to:

  • Create meeting agenda for weekly team meetings that I led
  • Write weekly status reports that were sent to my supervisor and client
  • Set up sprints on JIRA and track story progress to ensure efficient completion of tasks
  • Attend weekly team lead meetings and relay information to my team
  • Address and remove roadblocks that interns hit
  • Maintain effective communciation with our client to ensure our progress fit our client's needs

Design Process: For this project, our client already had a website that they liked, so we did not have to create any color schemes or page layouts. However, during one meeting, our client mentioned that she would love to show gardeners a preview of the project to promote the application. Thus, I had one intern use Adobe XD to create a prototype of the application for our client. This prototype helped us make iterations to how the client wanted the form and results page to appear that sketches could not quite convey.

We also had to design the database queries to only pull data that was necessary for the users. Our database has sixteen different columns attached to a single plant; all of this information is not directly revelant to the user. We concluded that a user would only need to see a plant image, common name, latin name, and a link to more information about the plant. We decided to omit plant colors, heights, and other descriptors because that is what the user initially inputted to find the plant, therefore they know that the plant fits that criteria. This removed a lot of excess clutter on the results page, making it easier for gardeners to find the plants they want.

Development Process: We created the INPAWS databse using MySQL. The list of plants that INPAWS had was in nan Excel sheet that did not support importing data into MySQL efficiently. Once our team edited the Excel entries, we imported it into our database that we set up. After discussing the database and our prototype with our client, we decided to make a few changes to it. This includes setting up the search queries to appeal to a range of heights for plants rather than adjectives describing various heights. The most challenging part of creating the database was inserting images to appear on the results page. After researching various options for adding the images, we decided to insert them as BLOB images and then pull the images using select statements.

We also developed an admin panel for INPAWS that allows for simple manipulation of the database. This allows for the administrator to add, edit, and remove plants from the database. When clicking on a plant to edit or adding a new plant, the setup is similar to how it appears on the website; the administrator can check boxes of criteria that matches the plant and input various heights, notes, or an image. We also created the plugin to have its own tab on the WordPress dashboard to clearly seaparate the application from the rest of the website.
